Angular 框架选型总结

发表于 2021-04-08 分类于 Angular
工作以来使用了几年的 Angular，从 Angular 2 Beta 开始，用到了现在的 Angular 11，最近工作变动，下一份工作里大概是会使用其他框架了，正好在此对 Angular 的选型和实践做一个总结。

在工作经历上，转向前端是 16 年，那时候的前端还是 jQuery 的时代。从 WitPM 项目开始立项后，Team Leader 选型了 Angular 2，这个时候的它还处于 Bate 末期，即将进入 Release。他选择这个框架的理由，应该是考虑到我们是从 C# 后端开发，转换为前端，这个框架的各种特性，相较于 React，对后端的同事更加友好（那时候的 Vue 还在襁褓之中）。

在之后进入知行公司，使用的框架是 AngularJS，也就是经历重构前的 1 代版本，和 Angular 2 大相径庭，后续的框架升级，也就优先选择了 Angular 2。

再到了 17 年开始，前端开始进入一个爆发期，大量的培训机构和新人涌入，因为企业的需求极大（老项目更新，新项目技术换代，行业发展），Angular 2+ 因为概念太多，学习曲线陡峭而被国内抛开，Vue 开始大热。我们也很自然的在 “天津市场监管平台” 这个项目中开始使用 Vue 来开发项目。经历整个项目6个月的开发，完成了项目的交付，也在使用中，感到一些局限性，所以在 18 年开始，偏大型，人员复杂的市场监管项目，以及共享交换平台等，又回到了 Angular 框架中。

这里主要总结一下具体的情况，以及面临的优劣之处，取舍考量。

在这之前，先介绍一下当时的团队情况。在 17 年我进入知行之后所见，这里的前端开发是比较粗放式的，虽然项目框架搭建得非常精致（陈师兄赛高！），框架方面在当时那个 jQuery 为王的时代，也是难得的使用了 AMD（dojo） 模块化工具，AngularJs 作为视图框架，Webpack 打包。但是在项目开发上，还是有一些问题，具体体现在：

  • 缺少开发流程规范：项目开发都是直接向 SVN 主分支提交，大家各自负责独立的某一个业务功能，没有代码审查（每个同事业务开发都饱和），没有相互交流，都是埋头完成自己模块。
  • 项目耦合度高：因为各自都是在自己的模块完成项目，缺少底层设计，公共资源混乱，有需要谁都可以在各个层级添加公共字段，维护各种状态。存在字段冲突，或者资源重复的问题。
  • 代码风格混乱：因为缺少审查，各自在独立的模块开发，项目风格上比较混乱，比如几个常见的情况：所有样式都写在 style 属性中、组件功能复杂，几千行代码的组件比比皆是、命名随意(a,b,吃temp1,temp2……)、变量类型混用等等。
  • 基础资源缺失：公司的项目主要分几个领域，每个领域内多个项目同质化程度很高。但每次同类型的新项目启动，都是直接把老项目复制一份，在那基础上改。冗余大量不需要的代码，谁也不知道某一部分是用来作什么的，干脆都留着。缺少跨项目的基础资源共享。

长期积累，每个项目都处于一种杂乱脆弱但又还勉强正常运行的 “危房” 状态。这些问题通过代码审查，规范流程，或者定期重构可以改善和解决，但是外包公司的工期限制，以及人员分配的惯有规则决定了此路不通。在这个模式下，更换框架（AngularJs -> Vue）的全新项目 “天津市场监管平台” 并没有得到好转，Vue 的灵活和轻量，在天津项目中依旧延续了以上问题。主要存在公共状态混乱，组件复杂庞大难以测试维护，模块间耦合度高的问题。

人员的工作流程上的改进行不通，就只有从项目实施上着手，最后总结下来主要想要解决几个问题：

  • 各个小组的模块之前，尽量完全隔离，任他内部翻江倒海，其他模块都不受影响。
  • 要使用 Typescript，对类型严格限制，尤其是限制函数，减少潜在的类型错误。
  • 让组件变轻，拆分业务组件，把UI交互逻辑，和具体的业务逻辑分离，组件本身只保留 UI 所需字段和函数。而具体的业务功能，及其子功能，分到一个个独立的包中，并在不同的组件中共享。
  • 建立基础模块，维护全局功能和状态，让各小组的模块可以共享，各个具体业务模块暂时先不管，但是基础模块要能够跨项目共享，作为同质型项目的基础，做到完全解耦，完全测试。

于是在 18 年开始，公司从江西瑞昌市场监管项目着手，又从新启用 Angular，和天津项目类似，开始重新搭建项目结构。

typescript 约束

这是当时选型 Angular 的重要因素，在那时对 TS 支持最完美的就是这个框架，毕竟它本身就是用 TS 构建的，并且特别激进的只允许用 TS 开发。
在实际使用中，比较明显的改善在于更好的描述性，边际情况的规避，类型提示的便捷。

  • 变量声明规范：
    工作强度一高，”temp1“、”a1“ 这种变量的出现就很常见，尤其是公司每年固定十几个应届生，几乎不经培训，就会直接分到各个项目组。新老成员都在各自的任务里自顾不暇，这些变量写完之后，不到一个月，自己还能不能认识它，就全指望注释了。
    TS 配置里规定不允许使用 ”any“，限制变量的使用，特别是对于对象类型，通过类型声明，知道它们的所属字段。
