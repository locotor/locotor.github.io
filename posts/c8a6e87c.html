懂车帝PC站从上手到上线

发表于 2021-06-29
加入懂车帝团队2个月了，之前很少接触 toC 的互联网项目，所以对开发到上线的流程还是有些陌生和茫然的，加之陌生的技术栈，带来了不小的熟悉成本，这里记录一下开发历程

熟悉开发工具

PC站项目需要更快的用户相应速度，和对SEO的良好支持，采用了 React SSR 方案，框架采用 Next.js + Koa + typescript 来实现。

在开发工具的知识准备上，需要熟悉：

  • React：项目主要采用 hooks api，只有少量的旧版代码存在一些 class component；
  • Next.js: 主要用到了服务端渲染，路由，组件的数据获取等；
  • typescript：加入了静态类型检测；
  • tailwindcss：简化样式编写，因为PC站要针对几个特定分辨率节点做响应式的布局，它可以帮助简化工作；

了解项目结构

打开一个陌生的前端项目，个人偏好从配置文件开始，可以看到配置了哪些功能和依赖，会比较容易弄清项目的结构。

  • package.json：首先从 package.json 开始，抛开庞杂的依赖配置，从 scripts 中可以看到，dev:XXX 开头的，都是开发状态的本地服务运行，冒号后面的是各种常用 ppe 环境参数。查看这些 dev 的脚本可以看到，都使用了 nodemon 自动检测目录中的文件更改，并通过重新启动应用程序来实时更新。
    nodemon 的具体配置在下方 nodemonConfig 中，里面配置了监听路径，以及执行命令行。从命令行中也可以看到服务器端的 ts 配置文件，以及路由IP、TCE_PSM。
    而通过 scripts 中的其他几个脚本可以看到，项目配置了 ts 类型检查，eslint 语法检查，prettier 格式化代码。并且配置 husky 工具，在 Git 提交时自动执行脚本，确保代码符合规范，并保持风格统一，防止因为代码格式化不一致，导致大量的 Git diff。

  • tsconfig：因为使用 Typescript 作为开发语言，所以根目录存在 tsconfig.json 和 tsconfig.server.json 两个配置文件，分别对应本地和服务器端配置。服务器端的 ts-node 配置继承自 tsconfig.json，只是在这基础上针对服务端node环境覆盖部分配置。
    需要注意的是，这里配置了各个路径的别名（否则编辑器会报错）。通过它们，既可以清楚地定位项目中核心的几个文件夹是哪些了。

  • tailwind.config.js：在这里可以自定义项目中用到的 tailwind 样式，例如配置了前缀为 “tw”，以及各种用到的距离，颜色，响应式临界值等，在实际开发中有需要可以在这里继续添加配置。

  • next.config.js：最后是 Next.js 框架的配置文件。里面第一个配置 useFileSystemPublicRoutes:false 就是禁用了服务端的文件路由，因为项目集成了 Koa 来处理服务的功能，具体详情可以参见另一篇 “Next.js-Koa-构建服务端渲染应用”。此外就是一些 webpack 的配置项（loader，路径别名，插件，腻子脚本）了

除了这些，还有一些如 prettier，eslint 的配置，主要是为了团队的代码规范，风格统一，就不一一列举了。
通过这些配置文件，可以看到项目的一个大体脉络。

  1. 项目使用 typescript 作为开发语言，集成 koa 服务器框架，在 node 环境执行服务端渲染页面。
  2. 提交 git 会触发代码检查和自动格式化，如果检查不通过，会阻止提交。
  3. 业务代码集中在 pages-components(旧版)和 new（新版）中，通用资源在 libs 和 components 中。以及其他的 static（静态资源）、constant（常量）、style（样式）。

了解开发约定

随着项目的迭代和测试，有了一些常用的约定，可能不会在需求中明确说明，但是会在每次测试的时候重点检查，收录如下：

  • 卡片的点击跳转：卡片的点击跳转为全区域可用，所以通常卡片会用 a 标签包裹。此外，要注意卡片中的按钮点击，要阻止事件默认行为（光阻止冒泡不够），以免受卡片链接影响发生跳转。也要防止埋点的时候触发了卡片本身和卡片内按钮的点击上报。
  • 两倍图，图片兜底图：全站使用的图片都要使用两倍图，并且需要在图片加载失败或者加载时使用兜底图，这个可以使用统一的组件：Components/view-image。只需要传参 “autoPixelRatio” 即可自动两倍图。传参 cusDefaultImg 即可实现默认兜底图。
  • 作者v标：所有作者头像显示的地方，都要有v标显示，这个可以使用 Components/author/author-avatar 组件，传递 userVerified（是否加v） 和 verifyType（蓝v、黄v……） 参数。
  • 封面图 hover 放大：同样是使用 Components/view-image 组件，并传参：hoverTransition。
  • 字段过长省略处理：有全局的 class 类名（line-1、line-2……），在限制元素宽度下，就会生效。
  • 滚动条：带滚动条的元素，是在鼠标 hover 到元素内后才显示，可以使用全局类名 g-cus-scroll-bar。
  • 跳转：PC 站的跳转是左侧导航区点击，则当前页面切换新页面，而在页面中的链接，都是打开新页面，这是为搭建更符合 pc 端用户习惯的路径，降低页面跳出率。

可能还未能覆盖所有的约定，后续再持续更新。

发布测试环境

模块的测试通常是在一个特定的 ppe 环境下进行。刚加入开发，通常不需要自己搭建一个 ppe 环境，而是合入一个已有的。

如果用过 Jenkins 或者 GitHub Action 就会很熟悉这个流程。只需要将当前的开发分支，合入一个特定的 Git 分支，并推送到远程服务器上，就会触发编译发布流水线。

这个时候需要手动卡点确认一下，然后就等待发布上线，再进行测试即可。

需求评审&功能提测

每次需求开发前，会有产品原型进行需求粗评环节，UI 稿完成后会有需求细评环节，并会确定一些关键的提测准入功能点。在开发完成后，进入提测前，需要开发按准入文档，自测关键功能点，确保准确无误再提交。

上线功能

最后，终于进入激动的上线时刻了。需要从自己的功能开发分支，提交 merge request 到 master 分支，并邀请其他同事进行 code review。确保代码没有问题后，合入 master 分支。

和发布测试环境类似的，会触发编译流水线。不同的是，编译完成后，会先上线 ppe-prod 环境，并进入第二个人工卡点，这时候需要测试同学再次进行测试和回归，无误后才点击通过卡点，进入小流量，TCE卡点，最后上线。
